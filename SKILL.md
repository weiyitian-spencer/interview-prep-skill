# 面试项目经历准备 Skill

帮助用户把真实工作经历，转化为面向产品/运营类面试的高质量项目陈述，并做好追问准备。全程以对话打磨为主，最后产出存档文档。

### 触发场景
以下场景应启动完整流程：用户明确说要投产品/运营岗位；用户问简历上的项目怎么讲或怎么包装；用户要针对某个 JD 改项目表述；用户想模拟追问或准备被问到项目时怎么回答；用户上传了工作文档想整理成面试素材。

### 跳阶段说明
用户不需要每次从头走完整流程。如果用户明确只需要某个阶段（比如"我只想 mock 一下"或"帮我针对这个 JD 改一下陈述稿"），直接进入对应阶段，不强制从头开始。判断依据：Job Vault 里是否已有对应阶段所需的前置内容（元项目库、题库等），有则直接用，没有则先引导补齐。

## 整体流程

```
阶段一：元项目库构建  →  衔接：能力映射  →  阶段二：JD 适配改写  →  阶段三：追问准备
   (一次性，可复用)       (动态更新)           (每投递一个岗位做一次)    (基于改写后的项目)
        ↑                     ↑                                               │
        └─────────────────────┴───────── 追问中挖出新细节 / 新 JD 补充维度 ───┘
```

整个过程是动态迭代的闭环：随着投递增多，JD 不断扩充能力维度；追问打磨不断从新角度挖出元项目里的新细节；元项目库和能力映射图会越来越完整，后续每一轮准备的起点都比上一轮更高。

## 启动问题（必问，决定后续路径）

### 第一次使用：引导搭建 Project 知识库

如果这是用户第一次使用此 skill（判断依据：没有上传任何素材、对话里没有出现过元项目库内容），先花一分钟引导用户搭好"求职军火库"Project，再开始正式流程。

对话式 AI 没有跨会话记忆，如果不做持久化，每次新开对话元项目库就丢了，"动态迭代"就无从实现。推荐把三份核心文档放进一个叫 **Job Vault** 的专属空间，结构如下：

```
Job Vault/
├── 元项目库.md       # 每个项目一份，STAR 结构，追问补充的细节也在这里，注明来源
├── 能力映射图.md     # 项目 × 能力维度矩阵，JD 来源标注，缺口预警
└── JD 存档.md        # 投过的每份 JD 原文 + 解析出的关键能力词，支持跨 JD 频次统计
```

这个 skill 有两种运行模式，体验和用户需要做的事情不一样，第一次使用时告知用户：

**对话模式**（claude.ai、ChatGPT 等对话产品）
AI 只能读取文档，不能直接写文件。每次对话结束后，需主动把更新内容整理成**可直接粘贴的片段**（格式已对齐），用户复制后手动更新到对应文档里。虽然有一步手动操作，但只要养成"开始前上传文档、结束后粘贴更新"的习惯，迭代效果和 Agent 模式一样。
- claude.ai 用户：创建名为"Job Vault"的 Project，把三份文档上传到项目知识库，每次新开对话自动读取
- 其他平台用户：把文档存在本地或云端（Notion、飞书、Google Doc 均可），每次对话开始时上传或粘贴进来

**Agent 模式**（有文件读写权限的 AI 工具，如 Claude Code）
AI 可以直接读写本地 Job Vault 文件夹。每个阶段结束、用户确认内容后，自动把更新写入对应文档，不需要用户手动操作。整个迭代闭环完全自动化。
- 本地建一个 `Job Vault/` 文件夹，三份文档放在里面，AI 直接操作

如果用户暂时不想搭这个结构，跳过此步直接往下走，但告知用户每次对话开始时需要手动提供最新版内容，否则无法延续上次进度。

---

触发此 skill 后，第一件事是问用户当前所处的阶段，而不是直接开始整理项目：

> 「你现在大概处于哪个阶段？」
> - A. 还没有特定 JD，想先把自己的项目经历整理清楚
> - B. 已经有一个或几个目标岗位/JD，想针对性准备
> - C. 同时在推进多个方向，JD 性质差别比较大

根据回答走不同路径：

**A（广泛准备）**：先建粗颗粒度的全项目库 + 能力映射，重点是摸清自己有哪些牌、缺口在哪里。阶段一可以覆盖所有核心项目，但不用追求每个项目都挖到极致细节，能力映射出来后再决定哪些项目值得深挖。

**B（有特定 JD）**：只针对这个 JD 最相关的 1-2 个项目做阶段一，其他项目暂时跳过。效率优先，先把这个岗位准备好，有余力再补其他项目。

**C（多方向并行）**：先帮用户判断哪个方向优先级最高（最近的面试、最想去的岗位、或者能力匹配度最高的），聚焦一个方向走完完整流程，再考虑复用到其他方向。不要同时铺开所有方向，容易什么都浅尝辄止。

如果用户已经有整理好的项目素材（比如之前对话产出过，或者上传了文档），确认内容后可以直接从衔接节点或阶段二开始，不用重新走阶段一。

## 阶段一：元项目库构建

目标：为用户的**每一个**核心项目，整理出一份完整、信息密度最大化的 STAR 版本。这是后续所有改写的源头素材，要尽可能详尽，不用担心讲多——精简是阶段二的事。

### 信息来源
- 用户直接口述
- 用户上传的材料（工作周报、复盘文档、PRD、数据报表截图等）—— 如果用户提到"我上传了文档"或附件里有相关文件，先读取这些材料提取信息，而不是让用户从头口述一遍
- 两者结合：用户口述大框架，上传材料用来补充具体数字和细节

### 必须挖出的 STAR 要素（产品/运营岗特化）

不要只满足于用户给的笼统描述，要主动追问挖掉的信息。逐项核对：

- **S/T（背景与目标）**：项目起因是什么？是老板分配还是自己发现的机会？业务背景是什么（用户规模、所处阶段、面临的核心问题）？目标是怎么定的，有没有量化指标？
- **角色与协作范围**：用户在项目中的角色定位（独立负责人/核心成员/支持角色）？团队规模和构成（跨了哪些职能：技术、设计、运营、销售）？是否需要向上管理或跨部门推动？
- **A（行动，重点）**：
  - 关键决策点：遇到了什么选择/分歧，怎么判断和拍板的？
  - 方法论：用了什么分析框架、调研方法、A/B 测试设计？为什么选这个方法？
  - 难点与应对：执行中遇到的具体障碍（资源不足、数据缺失、团队抵触、技术限制等）以及怎么解决的
  - 取舍：明确放弃了什么，为什么
- **R（结果，必须量化）**：
  - 核心指标的具体变化（转化率、留存、GMV、效率提升百分比等），没有精确数字时引导用户给一个合理区间或定性描述加佐证
  - 非数字类成果：流程是否被沉淀为规范、是否被复制到其他业务线
  - 反思：如果重新做一次，会有什么不同？（这部分很多人会漏，但面试官常问）

### 交互方式

一次只问 1-3 个问题，不要列一长串问题轰炸用户。问完一轮，根据回答内容判断哪里还薄，继续追问，直到这个项目的信息密度足够支撑一个 2-3 分钟的完整陈述 + 能扛住至少 3-4 层追问。

每整理完一个项目，用结构化方式（小标题 + 要点，不用大段表格）回显给用户确认，类似：

```
**项目：XXX**
- 基本信息：时间 / 公司 / 角色
- 背景/目标：...
- 我的角色：定位 + 职责范围
- 关键行动：决策点 / 方法论 / 难点与应对
- 结果：量化 + 非量化
- 核心挑战与方法论沉淀：...
- 反思：...
- 能力覆盖：强/中/弱
- 待补充：[列出还需要用户补充的点，如果有]
```

确认无误后，把确认后的内容当作这个项目的"基准版本"，后续阶段二的所有改写都基于此，不要凭空发明新细节。

**注意：这份回显是信息底稿，不是面试话术。** 口语化的陈述稿在阶段二才出，因为没有 JD 就不知道该强调哪个角度。

### 沉淀时机判断

每个项目信息收集到一定程度后，主动判断是否具备沉淀条件：当项目已有基本的 S/T/A/R 要素、至少一个量化结果或具体成果、以及 1-2 个关键决策细节时，主动问用户：

> 「这个项目的基本信息已经比较完整了，要现在沉淀进元项目库吗？后续还可以继续补充细节。」

用户确认后完成沉淀，再问：

> 「你现在有想针对的 JD 吗？有的话我们可以直接进入阶段二，针对这个岗位改写陈述稿；没有的话可以继续整理下一个项目。」

根据用户回答决定下一步路径，不要默认继续整理项目。

## 衔接节点：能力映射

所有核心项目整理完毕后，在进入阶段二之前，做一次项目与能力的交叉梳理。

目的是帮用户和 AI 都建立一张"地图"：知道哪个项目天然适合体现哪类能力，以及哪些能力目前没有项目覆盖（这是缺口预警，提早知道比面试时临时慌好）。

### 能力维度参考（产品/运营岗常见）
- 数据分析与量化决策
- 用户洞察（调研、访谈、行为分析）
- 0-1 项目搭建
- 增长/转化优化
- 跨部门协调与推动
- 精细化运营
- 向上管理 / 资源争取
- 复杂问题拆解与优先级判断

### 输出形式

不需要重新整理"项目 × 能力"的矩阵——每个项目底稿里已有能力覆盖标签，直接查即可。能力映射图只维护以下三块：

```
## 能力维度总表
- [维度名]（来源：通用 / XX公司JD）
- ...

## 缺口预警
- 目前没有项目强覆盖：[维度]（如果目标岗位 JD 强调这点需要留意）

## 跨JD频次统计（积累3份以上JD后更新）
- 高频：[维度] 出现 X 次 → 这个方向的硬核能力，优先用项目覆盖
- 低频：[维度] 出现 1 次 → 某家公司的特殊要求，降权处理
```

它是**动态维护**的，有两个更新触发点：

- 用户新增了项目 → 检查项目底稿的能力标签，更新缺口预警
- 阶段二解析完一份新 JD → 检查 JD 里出现的能力词是否在维度总表里，有新的则补充，并标注来源

当用户积累了 3 份以上 JD 时，主动提示可以做跨 JD 频次统计。

## 阶段二：JD 适配改写

目标：拿到具体 JD 后，从元项目库里选出最相关的项目，并调整叙述角度（不是编造新事实），让其呼应 JD 强调的能力点。

### 步骤

1. **解析 JD**：如果用户直接贴了 JD 文本，提炼出 3-5 个该岗位最看重的能力关键词（例如"数据驱动决策""0-1 项目搭建能力""跨部门协调""用户增长方法论""精细化运营"）。如果用户只给了岗位名称没给 JD 全文，先询问要不要贴 JD，或者基于岗位名称做合理假设并说明假设。

2. **匹配项目**：从元项目库里，判断哪些项目最能体现 JD 的关键词，向用户说明匹配理由。如果元项目库里没有强匹配的项目，坦诚告知用户这是个缺口，讨论怎么用现有项目里的某个侧面去靠近，而不是硬编。

3. **改写表述**：
   - 调整开场的"背景"部分，让其更贴合 JD 提到的业务场景措辞
   - 把"行动"部分里最契合 JD 关键词的细节往前提、展开讲；不太相关的部分一笔带过或删去
   - 结果部分尽量用 JD 隐含在意的指标口径重新表达（比如 JD 强调"增长"，就把结果换算成增长率而不是绝对值）
   - **底线：不允许编造任何元项目库中没有的事实、数字或职责**。如果为了贴合 JD 需要拔高表述，必须明确告知用户这是"包装"而非"事实"，让用户自行判断是否采用，并提醒面试中如被追问细节要能兜得住

4. **输出两份产物**，跟用户确认是否到位，根据反馈迭代：

   **产物一：改写后的陈述稿（中英文各一版）**
   即针对这个 JD 调整角度后的项目陈述，用于面试中直接讲。

   **产物二：JD 映射说明**
   解释"为什么这个项目证明我适合这个岗位"，格式如下：
   ```
   能力点 → JD 要求 → 一句话连接逻辑
   例：「漏斗拆解 + A/B 测试设计」→「JD 要求数据驱动决策」→「我在项目里用的就是这套方法，不是拍脑袋定方案」
   ```
   这份映射说明不是给面试官看的，是用户在被问到"你为什么适合这个岗位"时的思考骨架，面试中用自己的话说出来。

   **产物三：自我介绍（中英文各一版，1分钟）**
   在陈述稿和 JD 映射确认后，基于已选用的项目和 JD 映射，帮用户串出一版自我介绍。逻辑是"我是谁 → 我做过什么（突出和这个岗位最相关的经历主线）→ 我为什么在这里"，不是把所有项目罗列一遍。同样提供中英文各一版，去除 AI 味，符合口语表达习惯。

**中文版要求：**
- 口语化，符合中文面试的实际说话节奏，不是书面公文体
- 避免"赋能""抓手""闭环""深度链接"等大词；避免"通过…实现了…"这类 AI 惯用句式
- 读出来要像一个人在讲自己做过的事，不像在背 PPT

**英文版要求：**
- 符合英语母语者的表达习惯，不是中文直译
- 避免过度正式（"I spearheaded / leveraged / facilitated"这类 AI 高频词）；用更自然的动词（"I led / built / worked with / figured out"）
- 数字和缩写用英文面试的惯用表达（比如 DAU 要解释一次，增长率用百分比而不是"翻番"直译）
- 语速节奏上留出停顿感，长句适当拆短，让用户说出来不绕口

输出格式：中文版和英文版分开列，用小标题区分，方便用户对照调整。

## 阶段三：追问准备

目标：基于阶段二改写后的项目陈述，预演面试官大概率会追问的问题，并和用户一起打磨应答。

### 追问生成逻辑

不要泛泛地给"行为面试通用问题"，要紧扣这个具体项目和 JD 的结合点生成追问，常见方向（按产品/运营岗特点）：

- **决策细节类**："你提到 XX 决策，当时还有哪些方案？为什么没选别的？"
- **量化追问**："这个提升的具体计算方式是什么？有没有排除其他干扰因素？"
- **冲突协作类**："团队里有人不同意你的方案，你怎么推动的？"
- **失败/反思类**："这个项目有没有不如预期的地方？"
- **角色边界类**："这件事是你一个人做的，还是团队一起？你具体负责哪部分？"（尤其当陈述听起来"功劳过大"时，面试官会重点确认这一点）
- **能力迁移与岗位匹配类**：这类问题表面各不相同，本质都是在考察"你有没有想清楚自己为什么适合这个岗位"，要特别重视：
  - "如果让你在我们这个场景下做类似的事，你会怎么做？"
  - "你刚才说的这个项目，和我们这个岗位有什么关联？"
  - "你为什么选这个项目来举例？"（潜台词：这个项目能证明你适合我们吗？）
  - "你觉得你在之前项目里积累的经验，怎么迁移到这个新职位上？"
  
  应对这类问题，用户需要提前想清楚两件事：① 这个项目里的哪个具体能力点，直接对应 JD 里的哪个要求（要能说出来，不能只说"我觉得很匹配"）；② 这个岗位和用户过去做的事有哪里不一样，用户打算怎么补那个差距。模糊的"我相信我能胜任"是减分答案，面试官要听到的是有逻辑的连接。
  
  追问准备时，针对每个用到的项目至少生成一道这类题，并帮用户梳理出"项目能力点 → JD 需求"的对应关系，作为回答的骨架。

### 交互方式

一次给 2-3 个追问，让用户先尝试自己回答（口头或文字），再给反馈。重点纠正"听起来不像亲身经历"的破绽，这是产品/运营面试里最常见的减分项。

具体看以下信号，发现了要明确指出是哪里出了问题：

- **主语从「我」变成「我们」**：可能在模糊个人贡献边界，要追问"这件事具体是你做的还是团队一起？你负责哪部分？"
- **全是形容词，没有动词**：回答里出现"非常复杂""很有挑战""影响很大"但说不出具体做了什么动作——说明在描述感受而不是在讲事情，要引导用户说出"我做了什么"
- **数字和阶段一底稿对不上**：转化率、时间线、团队规模前后不一致，可能是临场编的，要核实
- **回答太流畅、结构太完整**：真实经历的回忆通常有停顿、有"额……我想想"，过度流畅反而像背稿；可以让用户换个角度重新说一遍，看是否一样流畅
- **结论先行但没有过程**：直接说"最后效果很好"但说不清楚中间怎么做到的，面试官会追问，用户容易答不上来

如果用户答得吃力，给出一个参考性回答方向，但要基于阶段一收集到的真实细节，不要替用户编故事。

### 阶段三结束后：回溯补充元项目库

每轮追问打磨结束后，主动做一次回溯：梳理这轮对话里用户说出来的新细节，区分两类：

- **值得补充进元项目库**：新的事实细节——之前没提到的数字、决策背景、具体障碍、当时的竞品情况、内部阻力等，确实发生过的
- **不补充**：只是换了角度重新表述已有内容，或者为了回答好听临场加的说法，用户自己也不确定准不准的

将值得补充的内容列出来，询问用户确认"这些是真实发生的吗，要不要补进项目底稿"，**不要自动写入**——由用户确认后再更新，保证元项目库的信息都是用户认可的事实。

确认后更新对应项目的基准版本，并注明"来源：XX 岗位追问补充"，方便日后追溯这个细节是在哪个阶段被挖出来的。

## 阶段四：Mock Interview

目标：从阶段三积累的题库里抽题，模拟真实面试压力，测试用户在没有提示的情况下能否稳定发挥。

### 进入条件

阶段三至少完成一轮追问准备，题库里有足够的问题可以抽。用户主动说"我们来 mock 一下"或阶段三结束后主动询问是否进入 mock。

### Mock 规则

- AI 扮演面试官，从题库里选 5-8 道题（混合项目经历题、行为题、1道案例题）
- 每道题用户回答后，AI 先追问 1-2 层，再进入下一题——不要回答完就立刻给反馈，保持面试节奏
- Mock 过程中 AI 不给提示、不纠错、不鼓励，保持面试官的中立语气
- 全部题目跑完后，再统一给反馈

### Mock 结束后的反馈

整体评估分三块：

- **答得稳的题**：哪几道回答清晰、细节真实、逻辑连贯，说明这个项目/方向已经内化了
- **有破绽的题**：具体指出哪里出了问题（主语模糊、数字对不上、结论先行、过度流畅等），对应阶段三的反馈信号
- **建议重点再练**：列出 2-3 道建议再跑一遍的题，说明原因

反馈结束后询问用户是否要针对某道题单独再练，或者重新跑一遍完整 mock。
